Operational outcomes

Lock the routing layer before adding more features.

This is the part buyers depend on. Fancy future layers can wait. Broken routing logic cannot.

Verdict Meaning Typical operator action Example reasons
SAFE Approve the payout. Continue the approved payout route. OK
BLOCK Stop before funds move. Re-check network, asset, address, or memo/tag. NETWORK_MISMATCH, ASSET_MISMATCH, INVALID_ADDRESS
REVERIFY Pause and confirm route details. Confirm venue, memo/tag, or destination context. DESTINATION_REQUIRES_MEMO_OR_VENUE_CHECK, DESTINATION_NOT_CLASSIFIED
TEST_FIRST Use a low-risk trial before full amount. Confirm the route and destination ownership first. DESTINATION_IS_CONTRACT_OR_APP, DESTINATION_IS_BRIDGE_ROUTER
UNAVAILABLE Do not infer approval from this response. Retry or escalate instead of auto-approving. DESTINATION_LOOKUP_UNAVAILABLE, REQUEST_FAILED

Versioning

Document the contract now, so future changes have somewhere to go.

The current API should be treated as v1. New optional fields can be added without drama. Breaking meaning changes belong in a new version, not in a surprise Friday deploy.

Safe in v1

Add optional fields, add extra trace metadata, or improve English explanation text without breaking policy routing.

Not safe in v1

Changing what a stable reason_code means, changing the meaning of a verdict, or silently removing required routing fields.

Buyer rule

Integrators should route on verdict, reason_code, and next_action — not on free-form summary text.
