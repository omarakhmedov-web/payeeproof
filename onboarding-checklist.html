Guiding rule

A narrow pilot beats a broad pilot almost every time.

Onboarding checklist

Two-sided setup checklist.

From the client

  • Business owner name
  • Company name
  • Primary email
  • Technical contact
  • Use case in one paragraph
  • Current flow type
  • Networks in scope
  • Assets in scope
  • Destination types in scope
  • Example error cases or edge cases
  • Desired verdict action
  • Notification route for alerts or webhooks
  • Planned pilot start date
  • Planned pilot review date
  • Success criterion agreed

From PayeeProof

  • Review intake for fit and scope
  • Confirm the pilot type
  • Confirm supported networks and assets
  • Confirm what stays out of scope
  • Issue pilot access or test instructions
  • Send welcome email
  • Provide support contact path
  • Activate the pilot in the tracker
  • Set weekly summary date
  • Set end-of-pilot review date
Fast-start rule

Do not start wide if one narrow flow can prove value faster.

One workflow

Choose one real approval or review flow instead of spreading the pilot across unrelated cases.

One pain point

Anchor the pilot around the mistake pattern or ambiguity that matters most operationally.

One success criterion

Agree how value will be judged before the first live checks begin.

Next step

Use this page as the kickoff checklist.

Once the checklist is complete, the pilot can move into live verification, weekly review cadence, and end-of-pilot decision.
